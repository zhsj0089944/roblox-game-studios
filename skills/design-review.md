## Phase 0: 解析参数

提取 `--depth [full|lean]`，默认 `full`。

- **`full`**: 完整审查 — 所有阶段 + 专家代理
- **`lean`**: 所有阶段，不启动专家代理 — 更快

---

## Phase 1: 加载文档

完整读取目标设计文档。读取 `DESIGN.md` 了解项目上下文和标准。读取相关设计文档（检查 `design/gdd/` 下的相关系统）。

**依赖图验证**：对于文档中列出的每个依赖，使用 Glob 检查其 GDD 文件是否存在。标记不存在的——这些是下游作者会遇到的断链引用。

---

## Phase 2: 完整性检查

根据设计文档标准检查：

- [ ] 有 Overview 部分（一段话摘要）
- [ ] 有 Player Fantasy 部分（预期感受）
- [ ] 有 Detailed Rules 部分（明确的机制）
- [ ] 有 Formulas 部分（所有数学公式和变量）
- [ ] 有 Edge Cases 部分（异常情况处理）
- [ ] 有 Dependencies 部分（列出其他系统）
- [ ] 有 Tuning Knobs 部分（可调数值）
- [ ] 有 Acceptance Criteria 部分（可测试的成功条件）

---

## Phase 3: 一致性与可实现性

### 内部一致性

- 公式产生的数值是否与描述的行为匹配？
- 边界情况是否与主规则矛盾？
- 依赖是否双向（其他系统是否知道这个系统）？

### 可实现性

- 规则是否足够精确，程序员可以无需猜测就实现？
- 是否有"模糊"部分缺少细节？
- 是否考虑了性能影响？

### 跨系统一致性

- 是否与现有机制冲突？
- 是否与其他系统产生意外交互？
- 是否与游戏的既定风格和支柱一致？

---

## Phase 3b: Roblox 特定审查

### 服务器/客户端分离

- [ ] 数值计算是否在服务器？
- [ ] 视觉效果是否在客户端？
- [ ] 是否有防作弊验证？

### DataStore 兼容性

- [ ] 数据大小是否 < 4MB？
- [ ] 是否考虑请求频率限制？
- [ ] 是否有 Session Lock 处理？

### 性能影响

- [ ] 物体数量是否在预算内（基准 800 + 55/人）？
- [ ] 是否有热路径性能问题？
- [ ] 是否有内存泄漏风险？

### RemoteEvent 设计

- [ ] 事件协议是否明确？
- [ ] 数据结构是否有文档？
- [ ] 是否有频率限制？

## Roblox 设计文档特殊要求

### 公式必须完整

```lua
-- ❌ 不可接受
"积分根据基础值和加成计算"

-- ✅ 可接受
积分 = base × (1 + itemBonus) × (1 + multUp) × (1 + evoTouch) × eventMult × vipMult

变量表：
| 变量 | 类型 | 范围 | 来源 | 描述 |
|------|------|------|------|------|
| base | float | 1-100 | GameConfig | 基础积分值 |
| itemBonus | float | 0-0.5 | 物品系统 | 物品加成 |
| multUp | float | 0-2.0 | 升级系统 | 升级倍率 |
| evoTouch | float | 0-1.0 | 进化系统 | 进化触摸加成 |
| eventMult | float | 0.5-3.0 | 事件系统 | 事件倍率 |
| vipMult | float | 1.0-2.0 | VIP 系统 | VIP 倍率 |

预期输出范围: 0.5 到 600
边界情况: 当所有乘数为最大值时，积分可能溢出 → 限制最大值
```

### 边界情况必须明确

```lua
-- ❌ 不可接受
"如果玩家死亡，处理死亡逻辑"

-- ✅ 可接受
场景: 玩家在 Boss 战中死亡
预期行为:
1. 玩家进入死亡状态（isDead = true）
2. 停止所有伤害计算
3. 播放死亡动画（1.5s）
4. 3s 后在出生点重生
5. 重生时血量为 100%（满血）
6. 重生后 2s 无敌时间
7. Boss 脱战回到巡逻状态

边界情况:
- 如果 Boss 在玩家死亡动画期间也死了？→ Boss 死亡优先，玩家仍然重生
- 如果多个玩家同时死亡？→ 独立处理，不互相影响
- 如果玩家在重生动画期间断线？→ 数据已保存，重连后正常状态
```

### 依赖必须双向

```lua
-- ❌ 不可接受
"Boss 战斗系统依赖积分系统"
（但积分系统文档中没有提到 Boss）

-- ✅ 可接受
Boss 战斗系统:
- 依赖: 积分系统（击杀奖励）、掉落系统（战利品）
- 被依赖: 成就系统（击杀计数）、排行榜系统（击杀排名）

积分系统:
- 依赖: 无
- 被依赖: Boss 战斗系统、升级系统、进化系统
```

### 调参范围必须有依据

```lua
-- ❌ 不可接受
| 参数 | 当前值 | 安全范围 |
|------|--------|----------|
| Boss 伤害 | 20 | 10-50 |

-- ✅ 可接受
| 参数 | 当前值 | 安全范围 | 极端值影响 | 设计依据 |
|------|--------|----------|------------|----------|
| Boss 伤害 | 20 | 10-30 | <10: Boss 无威胁；>30: 玩家秒杀 | 玩家血量 100，希望 3-5 次击杀 |
| Boss 血量 | 1000 | 500-2000 | <500: 太容易；>2000: 战斗过长 | 目标战斗时长 2-3 分钟 |
```

## 审查深度指南

### lean 模式（快速审查）
- 完整性检查
- 内部一致性
- 基本可实现性
- 输出报告
- 不启动专家代理

### full 模式（完整审查）
- 所有 lean 模式内容
- 启动专家代理（如果可用）
- 跨系统一致性检查
- 公式边界值分析
- 性能影响评估
- 输出详细报告
